cua-lark 参考内容
所属赛道：⻜书 AI 校园竞赛 · 质量工程与智能测试方向
课题一句话：构建一个 Computer-Use Agent（CUA），借助视觉多模态大模型，像真实用户一样
操作⻜书桌面客户端，完成跨产品的自动化功能测试与质量评估。

一、课题背景
1.1 CUA 技术趋势
Computer-Use Agent（CUA）是当前 AI 领域最具前沿性的研究方向之一。CUA 能够像人类用户
一样，通过"看"屏幕、"想"策略、"做"操作，实现对桌面应用程序的自主操控。 2025 年以来，
OpenAI CUA、Anthropic Computer Use、字节跳动 UI-TARS 等产品和模型相继推出，标志着这
一技术从实验室走向实际应用。

1.2 测试领域的痛点

传统 GUI 自动化测试高度依赖 DOM 结构、元素选择器或 Accessibility Tree。面对 UI 频繁迭代时
十分脆弱——一次按钮位置调整或文案变更就可能导致大量测试脚本失效。基于视觉多模态模型
的 CUA 方法，可以像真实用户一样通过视觉理解界面语义并执行操作，天然具备更强的泛化性和
鲁棒性，有望从根本上改变 GUI 测试的范式。

1.3 ⻜书：理想的测试场景

⻜书作为字节跳动旗下的企业级协作套件，涵盖 IM 即时通讯、日历、云文档、多维表格、视频
会议、邮箱等丰富的子产品线。⻜书桌面客户端（Electron 架构）的 UI 复杂度高、交互模式多
样、子产品间联动频繁，是 CUA 技术落地到测试领域的理想实验场。

二、课题目标
设计并实现一个 CUA-Lark 智能测试Agent ，核心能力包括：

视觉感知：通过屏幕截图识别⻜书桌面应用的 UI 元素、界面状态和布局结构
语义理解：理解自然语言测试指令，将其拆解为可执行的操作序列
自主操作：通过模拟鼠标点击、键盘输入、滚动等操作完成测试任务
4. 状态验证：操作执行后，通过视觉比对或语义判断验证结果是否符合预期
5. 评估报告：自动生成测试执行报告，包含操作轨迹、成功率、耗时等关键指标
三、技术要求与建议方案
3.1 系统架构设计（必做）
参赛选手需设计一个完整的 CUA 测试框架架构，建议包含但不限于以下模块：
模块 职责 技术举例
视觉感知层 屏幕截图采集、 UI 元素识别与定位 截图 API + VLM （ GPT-4o / Claude 3.5 Sonnet / Qwen-VL / UI-TARS ）
规划决策层 将自然语言指令拆解为步骤序列 LLM 推理 + CoT（Chain of Thought）规划
执行操作层 模拟鼠标、键盘等人机交互操作 PyAutoGUI / NutJS / 系统级 API
状态验证层 执行后截图比对，判断操作是否成功 VLM 语义比对 / 像素级 Diff / OCR 校验
评估报告层 汇总测试结果，生成可视化评估报告 自定义 Dashboard / Markdown 报告
3.2 核心功能实现（必做）
A. 基础操作能力•支持对⻜书桌面客户端的基础 GUI 操作：单击、双击、右键、拖拽、滚动、文
本输入、快捷键组合
• 支持多步骤复合操作的串联执行（如：打开聊天 → 搜索联系人 → 发送消息 → 验证发送成功）
B. ⻜书功能覆盖（至少覆盖 2 个子产品）
子产品 建议测试场景示例
IM 即时通讯 发送文本 / 图片消息、创建群组、 @ 提及、搜索消息记录、表情回复
日历 Calendar 创建日程、邀请参会人、修改日程时间、查看忙闲状态
云文档 Docs 创建文档、编辑文本内容、插入标题/列表、分享文档
多维表格 Base 创建表格、添加字段、录入数据、切换视图
视频会议 VC 发起会议、加入会议、开关摄像头/⻨克⻛
邮箱 Mail 撰写邮件、添加附件、发送与回复
C. 自然语言驱动测试 支持通过自然语言描述测试用例，例如：
•"在⻜书中创建一个名为'项目周报'的新文档，并输入标题'2026年Q2项目进展'"
•"打开日历，创建一个明天下午 2 点的会议，邀请张三参加"
•"在IM中搜索'测试群'，发送一条消息'Hello World'，并确认发送成功"

3.3 进阶能力（加分项）
进阶能力 说明
异常场景处理 自动识别并应对弹窗干扰、权限不足提示、加载超时等异常
跨产品联动测试 如 IM 收到日历邀请 → 跳转日历确认 → 返回 IM 验证状态
自愈式执行 操作失败后自动分析原因，尝试替代路径完成任务
测试用例自动生成 基于⻜书功能文档或录屏，自动生成结构化测试用例
混合定位策略 视觉识别 + Accessibility Tree / DOM 信息融合，提升元素定位精度
多轮对话编排 根据中间执行结果，动态调整后续测试步骤
录制回放 录制人工操作轨迹，生成可复用的自动化脚本
