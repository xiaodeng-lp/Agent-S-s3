## 2. 背景

项目目标来自飞书 AI 校园竞赛质量工程与智能测试方向。核心问题不是“做一个 bot”，而是“像真实用户一样操作飞书桌面端，完成跨产品自动化测试与质量评估”。

传统 GUI 自动化依赖 DOM、选择器或 Accessibility Tree，面对飞书桌面端这种 Electron 场景时稳定性不足。项目希望借助视觉多模态模型，把界面理解、动作执行、状态验证和评估报告串成闭环。

## 3. 产品目标

产品目标分为 5 项：

1. 视觉感知：识别飞书桌面端页面、元素和关键状态。
2. 语义理解：把自然语言测试描述转换为结构化测试任务。
3. 自主操作：执行点击、输入、滚动、快捷键和多步流程。
4. 状态验证：验证每一步及整条测试用例是否成功。
5. 评估报告：输出结构化测试结果和可读报告。

## 4. 目标用户

当前主要面向：

- 项目开发者
- GUI Agent 算法与测试框架开发者
- 竞赛评审或内部技术评审者

未来可扩展面向：

- 企业内部桌面自动化测试团队
- 需要半自动回归的产品测试团队

## 5. 非目标

当前阶段不追求：

- 飞书开放平台 bot 路线
- 企业应用 API 优先路线
- 全子产品一次性覆盖
- 全 OS 同时支持
- 完全自由对话式执行而无结构化约束

## 6. 当前范围

### 6.1 核心范围

- `Windows + 飞书桌面端 + GUI-first`
- 自然语言测试输入
- 结构化测试用例解析
- workflow 驱动执行
- step 级和 case 级验证
- 结构化测试报告输出

### 6.2 子产品范围

当前推荐覆盖顺序：

1. `IM`
2. `Docs`
3. `Calendar`

## 7. 用户价值

完成后，这个产品应当让团队获得三类价值：

1. 能快速验证飞书桌面端真实用户路径。
2. 能在 UI 频繁变化时保留比传统脚本更强的鲁棒性。
3. 能为每次运行产出可审阅的测试结果，而不是只知道“跑没跑完”。

## 8. 功能需求

### F1. 自然语言驱动测试

系统必须支持自然语言测试描述输入，并转换成结构化测试任务。

### F2. 基础 GUI 操作

系统必须支持：

- 单击
- 双击
- 右键
- 拖拽
- 滚动
- 文本输入
- 快捷键组合

### F3. 多步骤 workflow 执行

系统必须支持：

- 一个测试用例包含多个步骤
- 每个步骤有目标动作和断言
- workflow 可以显式 fallback 与重试

### F4. 状态验证

系统必须支持：

- step 级验证
- case 级验证
- 失败分类

### F5. 报告输出

系统必须输出：

- 结构化结果文件
- 可读报告
- 关键运行指标

## 9. 进阶需求

优先级从高到低：

1. 异常场景处理
2. 自愈式执行
3. 跨产品联动测试
4. 测试用例自动生成
5. 混合定位策略
6. 多轮对话编排
7. 录制回放

## 10. 里程碑

### M0 运行基线

- Agent 能稳定启动
- 能保存截图和日志
- 有固定 smoke case

### M1 单步验证

- 完成 5 个单步操作验证
- 仅验证截图 -> 识别 -> 单步点击 / 输入

### M2 流程串联

- 完成 1 个子产品的 3 条端到端流程
- 具备状态验证

### M3 多产品覆盖

- 至少 2 个子产品稳定可跑
- 向 IM / Docs / Calendar 各 2+ 用例靠拢

### M4 评估体系

- 输出结构化报告
- 自动统计成功率、耗时、步骤数

### M5 进阶能力

- 至少落地 1-2 项进阶能力

## 11. 验收指标

当前建议最小验收指标：

- 单步点击 / 输入稳定可复现
- 至少 1 个 IM workflow 高成功率运行
- 至少 2 个子产品可运行
- 每次运行产出 `summary.json` 和 `report.md`
- 失败可归因为识别、定位、操作、验证中的某一类

## 12. 需求映射

| 原始需求 | PRD 承接 | 当前口径 |
| --- | --- | --- |
| 视觉感知 | F1、F4、M1 | 以截图、页面状态、关键区域识别为主 |
| 语义理解 | F1、F3 | 自然语言先转结构化 `TestCase`，不直接自由执行 |
| 自主操作 | F2、F3 | 先覆盖基础 GUI 动作，再串联 workflow |
| 状态验证 | F4 | 必须同时支持 step 级与 case 级验证 |
| 评估报告 | F5、M4 | 每次运行都要沉淀结构化结果和可读报告 |
| 至少 2 个子产品覆盖 | 6.2、M3 | 当前优先级为 `IM -> Docs -> Calendar` |
| 多步骤复合操作 | F3、M2 | 用例模型必须支持 `steps[]` 串联 |
| 进阶能力 | 9、M5 | 仅在核心主链稳定后推进 |

## 13. 文档边界

- 本文档负责范围、价值、里程碑和验收，不承载详细模块设计。
- 规划层、定位层、workflow 结构等实现细节放在 `Technical Spec`。
- 模块输入输出、返回字段和契约变更放在 `Interface Doc`。
